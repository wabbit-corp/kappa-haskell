  provided: `stageDumps`, `incremental`, and the nonstandard
  `unicodeSourceWarnings`/`legacyCharAlias` names, which §T.4
  does not list among the portable capabilities), `x-` extension
  directives this harness does not support (§T.3: an unsupported
  `x-` directive makes the test unsupported, never silently
  ignored), modes and backends with no implementation here
  (`mode compile`/`mode analyze`, `backend` profiles other
  than `interpreter` — there are no code generators; §T.4 makes
  unmet `requires backend` unsupported and `scriptMode` is not
  implemented), incremental step suites (§T.7 presumes Chapter 34
  session reuse; the `incremental` capability is not claimed),
  and the structured-diagnostic/stage-dump assertions (§T.5.1
  payload/label/related/fix/suppression, §T.5.3) whose underlying
  record fields this implementation's diagnostics do not carry.

  malformed directive, or ill-typed directive argument is a harness
  error."* The corpus directives `allow_unsafe_consume`,
  `assertRunStdout`, `assertExecute`, `assertEvalErrorContains`,
  `assertParameterQuantities`, `assertDoItemDescriptors`,
  `assertInoutParameters`, and `assertContainsTokenTexts` appear
  nowhere in Appendix T (§T.4 configuration list, §T.5 assertion
  lists) nor anywhere else in Spec.md; they are another
  implementation's private, non-`x-`-prefixed directives, so the
  spec-mandated outcome is harnessError. (`assertEval` and
  `assertDiagnosticCodes` are likewise nonstandard but have an
  evident portable meaning; this harness implements them as
